# UUID 사용 이유

## 장점
- **분산 환경 고유성**: 중앙 시퀀스 서버 없이도 전 세계적으로 충돌 확률이 극히 낮다.
- **오프라인 생성**: 클라이언트/배치/마이크로서비스가 독립적으로 ID를 만들고 나중에 동기화 가능.
- **추적성**: ID에서 생성 시간(버전 1)이나 특정 노드를 추론할 수 있어 디버깅에 유용하다.
- **보안/노출 방지**: 증가하는 숫자 ID에 비해 레코드 수를 추정하기 어렵고, URL 추측 공격에 강하다.

## 단점 및 대응
- 16바이트 저장 공간 + 인덱스 크기 증가 → `UUID` 타입 지원 DB 사용, 혹은 128bit BINARY로 저장.
- 랜덤 분포로 인해 클러스터형 인덱스 조각화가 심할 수 있음 → 버전 1, 6, 7 처럼 시간 기반 UUID를 채택하거나 ULID를 고려.
- 가독성 낮음 → UI/로그에는 짧은 형태나 추가 메타정보를 함께 기록.

## 적용 가이드
- 멀티 리전/분산 시스템, 오프라인 생성이 필요한 엔티티에 우선 적용한다.
- PK로 사용할 경우 DB 및 ORM에서 UUID 기본키 생성 전략(`@GeneratedValue` with `UUIDGenerator`)을 명확히 설정한다.
- 시퀀스형 PK가 필요한 경우(정렬, 파티셔닝)에는 보조 키로 UUID를 저장해 양쪽 장점을 취한다.
