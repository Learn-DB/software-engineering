# 서비스 비용 체계 가이드

## 주요 모델
- **구독형 (Subscription)**: 월/연 단위 정액. 플랜별 기능, 사용자 수, 지원 수준을 기준으로 계층화한다.
- **사용량 기반 (Usage-based / Consumption)**: API 호출, 처리 건수, 스토리지, 시간 단위 등 실제 사용량에 따라 과금. 고객과 공급자 모두 성장에 비례해 비용이 변동.
- **하이브리드**: 기본 구독 + 초과분 사용량 과금. 예) SaaS 시트 요금 + 추가 API 호출 비용.
- **티어드 프리미엄**: 기본 무료/저가 티어로 유입 후, 고급 기능/보안/통합은 상위 티어에서 제공.

## 요즘 대세와 흐름
1. **Product-Led Growth + 사용량 과금**: 무료/저티어에서 사용 데이터를 수집하고, 가치 경험 이후 유료로 전환.
2. **단일 고객 ID 기반 번들링**: RevOps/FinOps와 연계해 여러 제품을 묶어 LTV를 확장.
3. **투명한 실시간 사용량 대시보드**: 고객 스스로 비용을 예측하고 최적화할 수 있는 Usage Metering 기능 필수.
4. **지속 가능한 단가 정책**: 인프라/라이선스 비용 변동을 반영해 자동으로 마진 관리(Band Pricing, Commit 할인)한다.

## 설계 체크리스트
- **단가 단위(Unit Metric)**: 고객에게 직관적인 지표인지 검증 (예: "학습자 좌석", "동시 세션", "API 1,000콜").
- **데이터 파이프라인**: 제품 사용량 → 청구 시스템 → 회계 자동화를 위한 이벤트 스트리밍 구축.
- **계약 옵션**: 선불 크레딧, 약정 할인, 초과 사용 알림 등 유연한 조건 제공.
- **옵스 연계**: RevOps/FinOps 팀과 협업하여 가격 실험, 손익 분석, 고객 세그먼트별 전략을 반복.

## 실행 팁
- Beta 단계에서 여러 가격 모델을 실험하고, 고객 인터뷰로 가치 지불 의사를 검증.
- 청구/결제 시스템(Fusebill, Chargebee, Stripe Billing 등)을 IaC와 함께 정의해 환경 간 일관성을 유지.
- 서비스 가격 변경 시 커뮤니케이션 플랜(사전 통지, FAQ, 마이그레이션 옵션)을 준비해 churn을 줄인다.
